# SuperT OTA升级说明

## 1. 当前项目的 OTA 方式

`SuperT` 当前使用的是涂鸦主联网固件 OTA 流程。

- 设备端通过固件 key 和版本号接入涂鸦云
- 升级包先上传到涂鸦开发平台
- App 侧负责提示用户、触发升级和展示进度
- 设备端再根据云端下发的升级信息自行下载升级包并执行升级

这不是“把本地 bin 文件先发到手机，再由手机把文件直接传给设备”的模式。

## 2. 当前工程里的 OTA 结论

当前 `SuperT` 没有注册“接管主控 OTA”的 `pre-upgrade callback`，因此主控固件 OTA 仍由框架默认处理。

当前源码中的 `gw_ug_cb` 只用于打印 OTA 元信息日志，不负责替换默认 OTA 下载与写入流程。

## 3. 编译后要拿哪个文件做 OTA

编译完成后，当前项目会在 `apps/SuperT/output/<版本号>/` 目录生成 3 类主要文件：

- `SuperT_QIO_<版本号>.bin`
  生产/整包烧录固件
- `SuperT_UA_<版本号>.bin`
  用户区固件
- `SuperT_UG_<版本号>.bin`
  OTA 升级固件

用于涂鸦 OTA 上传和发布的，应该是 `UG` 文件。

例如：

- `software/TuyaOS/apps/SuperT/output/1.0.1/SuperT_UG_1.0.1.bin`

## 4. OTA 的标准操作流程

### 第一步：编译一个更高版本的固件

设备当前如果上报的是 `1.0.0`，那你至少要编一个更高版本，比如 `1.0.1`。

版本号来源于构建参数里的 `APP_VERSION`，最终会传给设备的 `USER_SW_VER`。

### 第二步：找到对应版本的 `UG` 升级包

进入输出目录：

- `software/TuyaOS/apps/SuperT/output/<版本号>/`

拿出：

- `SuperT_UG_<版本号>.bin`

### 第三步：上传到涂鸦开发平台

在涂鸦开发平台中，一般按下面的顺序操作：

1. 进入目标产品。
2. 打开 `Product > Device > Firmware Management`。
3. 选择当前产品对应的固件。
4. 新建一个固件版本。
5. 上传 `SuperT_UG_<版本号>.bin`。

### 第四步：启用升级固件版本

上传完成后，需要把这个版本启用为 `Firmware Updates` 类型。

调试阶段建议：

- 先使用有限范围或白名单
- 确认测试设备已经在可见范围内

### 第五步：创建升级发布任务

再进入：

- `Product > Device > Firmware Update`

创建一次新的升级发布任务。

常见发布方式有：

- `Update notification`
  App 打开设备面板时提示用户更新
- `Forced update`
  强制升级
- `Check for updates`
  用户手动检查更新

## 5. 用户在手机 App 上怎么触发升级

当平台已经完成上传、启用和发布之后：

1. 让设备保持在线。
2. 用涂鸦 App 打开对应设备面板。
3. 根据发布方式：
   - 如果是提示型升级，App 会弹出升级提示。
   - 如果是手动检查型升级，可以在设备设置页或升级入口里检查更新。
4. 用户确认后，设备开始 OTA。

所以手机 App 的作用是“接收云端发布任务并触发升级”，不是“手动选本地 bin 文件上传给设备”。

## 6. 当前项目升级前需要满足的条件

- 设备已经成功配网并在线
- 上传到平台的版本号高于设备当前上报版本
- 上传的固件属于同一个产品/同一个 firmware key
- 设备处于当前升级发布的范围内

如果升级任务已经发布但 App 看不到更新，优先检查：

- 设备上报版本是否正确
- 新版本是否真的高于旧版本
- 固件版本是否已经启用
- 升级任务是否已经发布
- 测试设备是否在白名单/灰度范围内

## 7. 当前项目的 OTA 包模式

当前工程的 `app_resource_config.json` 中，`ota_mode` 配置为 `compress`，表示当前生成的 OTA 包走压缩升级模式。

## 8. 当前项目里和 OTA 相关的几个关键认识

- `output/T5_SuperT/` 目录里的内容主要是编译中间产物和静态库，不是拿给 App 上传的 OTA 文件
- 真正面向 OTA 上传的是 `apps/SuperT/output/<版本号>/SuperT_UG_<版本号>.bin`
- 当前 `gw_ug_cb` 只是日志回调，不等于你已经接管了主控 OTA
- 如果将来你想自己接管主控 OTA，才需要使用 `tuya_svc_upgrade_register_pre_cb(...)`
